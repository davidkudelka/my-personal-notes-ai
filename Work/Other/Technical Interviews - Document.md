  * # [SUGGESTION] Structure
    * General rule should be that before the interview the interviewers should decide who is responsible for each topic. This person would be leading asking questions regarding this topic - starting the conversation. This person should allow other interviewers to ask questions in those topics as well. We shouldn't jump from one topic to another, rather stick to one until it make sense.
    * ### Todo list before interview
      * List out with other people what topics should be discussed on the interview
      * For each topic there should be person responsible for it
      * Skim through engineer levels specified by OAK'S LAB - you will have to assign person to one level (this should give higher level idea about candidates level, don't take it too seriously)
    * ## How to ask questions?
      * It is always better to not specify technologies or name of libraries (it doesn't give interviewed person hints, he/she can sometimes surprise with mentioning unknown approaches, interesting points of view)
        * Good question
          * What tooling do you use when developing React applications?
        * Bad question (In my opinion gives interviewed person hint and he/she can just give very general answer without understanding importance of such library)
          * What do you think about eslint?
      * Ask about trade offs between different things (this let's candidate answer however he wants and it shows if he/she really understands the technologies - software development is always about trade offs)
        * Good question
          * What is the difference between GraphQL and REST and in what expects is one better than the other?
        * Bad question
          * Do you like GraphQL?
      * You can see the general pattern, question should always leave space for more flexible answer, these days we are using a lot of different approaches and libraries, most important is if candidate understands why he/she choose something and if he/she knows as well some alternatives (nothing is black or white)
      * ### Tips
        * If you are **Senior Engineer**
          * Ask about his past experience, problems he solved, issue that occured and how candidate learned from it
          * Try to be more technical and understand how candidates think about problems
        * If you are **Junior Engineer**
          * Try to think about what you expect from you senior engineer and then ask questions about it (How to solve different issue, how this person would help you - this is mainly about **Leadership capabilities** or **Motivations and interests**, try to find out what it would be like to work with this person)
    * ## Frontend
      * What do we consider important
        * **React**
          * Candidate should show basic understanding of React framework
          * Generally advised to ask about some React frameworks candidate knows - like Next.JS, CRA, etc...
            * Candidate should explain differences between those frameworks, trade offs, aspects
          * Advised to ask about state management tools - what tools are there and which he/she prefers and why
        * **General**
          * Differences between SSR, SPA, static generated
        * **A design systems, components libraries**
          * Ask about different design systems or components libraries candidates have experience
            * When does it make sense to use such library? Let him explain thinking behind it, trade offs, etc...
        * **Styling**
          * Differences between different approaches - discuss trade offs
    * ## Backend
      * What we consider important
        * **Databases**
          * Indexes
          * SQL vs NoSQL
          * ORM vs query builder vs raw sql
        * **Performance of the applications**
          * How to track performance - monitoring
          * What tools to user for monitoring
        * **Software architecture**
          * Discuss trade offs between different aprroaches
            * Monolith vs Microservices
            * GraphQL vs REST
            * Functional programming vs Object Oriented
            * Horizontal vs Vertical scaling
    * ## BACKEND / FRONTEND
      * **CI/CD pipelines**
        * What is candidates experience - discuss trade offs (We have it engineering levels)
      * **Project tooling**
        * eslint, prettier, typescript - it is important that candidates have basic knowledge about those tools
    * ## Leadership capability
      * This topic should be mainly focused on the leadership experience of the person
      * Topics
        * Communication skills
          * How is candidate solving communication issues in the team
        * Taking ownership of the different parts of system
          * Find out if candidate is taking ownership of something (We have this in engineering levels)
    * ## Motivation and interests
      * Find out what is the person most passionate about, to better understand his streghts and motivations
      * Ask about new technologies
        * We have it engineering levels
          * Is expected to actively follow engineering trends and propose changes, as well as analyse and feedback changes proposed by others.
