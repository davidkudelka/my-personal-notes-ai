    * **TDD eradicates the fear of change**
      * By implementing TDD you gain much better test coverage, which leads to 40% - 80% fewer bugs in production
    * **What is TDD?**
      * 1. step - Before you start implementation code, write test that proves if the implementation code fails or works (first run the test, to ensure it fails initially)
      * 2. step - Write the implementation and watch the test pass
      * 3. step - Refactor if needed. You should feel confident refactoring your code now that you have a test to tell you if you've broken something
    * **How TDD Can Save You Development Time?**
      * Initially it is hard to learn how to write testable code and how to test already written code, there is a **learning curve** (it takes 15% - 35% more time), but after some time, you can to the point where you actually write code much faster because of the tests
      * Test coverage can serve as well as documentation on how certain feature should work 
      * If you implement watch script over you tests, you can live time watch how your changes in code affects those tests
    * **TDD Taught Me How to Write Better Code**
      * It is something that can really teach how important is the separation of different parts of application, it can really teach you how important is to break things into smaller problem and then compose those smaller solutions into solution that solves bigger problem
    * **How TDD Saves Whole Teams Time?**
      * It removes the fear from merging and making change, it **let changes thrive**
